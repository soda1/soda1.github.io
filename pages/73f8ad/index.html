    线程不安全的集合

    探讨arraylist和hashmap 线程非安全的原因

    # Arraylist

    ArrayList底层是Object数组，在添加元素的时候没有使用锁，多线程操作时会出现预期之外的情况

    测试结果

    原因：

    ​ 当t1执行ensureCapacityInternal(size + 1)后让出时间片后，t2正在执行ensureCapacityInternal(size + 1)中让出时间片，t1恢复后继续执行，从而抛出ArrayIndexOutOfBoundsException

    # 解决方案

    1. 使用工具类的同步方法

      List<Integer> strings = Collections.synchronizedList(new ArrayList<>());
      1

      内部类SynchronizedCollection所有方法都是同步的。

    2. 使用并发包下的CopyOnWriteArrayList

      故名思忆就是写时复制

      /**
 * Appends the specified element to the end of this list.
 *
 * @param e element to be appended to this list
 * @return {@code true} (as specified by {@link Collection#add})
 */
public boolean add(E e) {
    final ReentrantLock lock = this.lock;
    lock.lock();
    try {
        Object[] elements = getArray();
        int len = elements.length;
        Object[] newElements = Arrays.copyOf(elements, len + 1);
        newElements[len] = e;
        setArray(newElements);
        return true;
    } finally {
        lock.unlock();
    }
}

      add方法在添加元素时会加入lock锁，并拷贝一份数组副本来添加元素，最后数组引用执行新的数组。

      加了lock锁却还要拷贝出一份数组副本的原因：

      ​ 如果有其他的线程在使用迭代器遍历数组时，修改数组元素会产生ConcurrentModificationException异常，这是Java的fast-fail机制。通过拷贝出来一份进行添加元素操作时，其他线程遍历数组也就不会出现异常了，而synchronizedList会。

    3. 性能比较

      由于CopyOnWriteArrayList添加元素时复制数组，所以如果写的元素过多时花费的时候会比synchronizedList要多得多，元素比较少时花费时会比synchronizedList少，下图为两种方案开两个线程进行写操作时花费的时间。因此CopyOnWriteArrayList适合于写操作比较少的场景。

          CountDownLatch syncCountDown = new CountDownLatch(2);
    CountDownLatch copyCountDown = new CountDownLatch(2);

    List<Integer> list1 = Collections.synchronizedList(new ArrayList<>());

    List<Integer> list2 = new CopyOnWriteArrayList<>();

    long start = System.currentTimeMillis();
    new Thread(() -> {
        for (int i = 0; i < 10000; i++) {
            list1.add(i);
        }
        syncCountDown.countDown();

    }).start();

    new Thread(() -> {
        for (int i = 0; i < 20000; i++) {
            list1.add(i);
        }
        syncCountDown.countDown();

    }).start();
    syncCountDown.await();
    long end = System.currentTimeMillis();
    System.out.println("synchronizedList time operation: " + (end - start));


    long start1 = System.currentTimeMillis();
    new Thread(() -> {
        for (int i = 0; i < 10000; i++) {
            list2.add(i);
        }
        copyCountDown.countDown();

    }).start();

    new Thread(() -> {
        for (int i = 0; i < 20000; i++) {
            list2.add(i);
        }
        copyCountDown.countDown();

    }).start();
    copyCountDown.await();
    long end1 = System.currentTimeMillis();
    System.out.println("CopyOnWriteArrayList write time operation: " + (end1 - start1));
}

        假设线程1、线程2都准备执行到tab[i] = newNode(hash, key, value, null)这一步赋值操作，都向着同一个bucket增加第一个元素，无论哪一个先执行都会出现数据覆盖现象。

      2. put和get同时进行并发时，可能会导致get操作未null

        在put操作进行resize时，会将old array里面的元素移到new array里面，在这过程中，get操作如果要获取的元素已经移到new array里面了，那么就会得到一个null值。

      1. 同1.8

        循环链表产生的原因是在transfer()方法中采用了头插法（后面的key放在数组开头）的方式插入到新数组里面去，从而在并发resize时导致。当线程1，线程2同时进行resize，线程1执行到Entry<K,V> next = e.next;时，由于时间片使用完了,线程1进入等待状态，这时候 e= [3,A] next=[7,B]。线程2 resize完成后 ，此时[7,B].next指向[3,A]，线程1获取时间片后继续执行，在执行插入完[7,B]后，由于next=[3,A]这时候会继续执行就会导致循环链表的产生。

    # 解决方案
    1. Collections.synchronizedMap()封装HashMap（所有方法都是同步的）
    2. 使用ConcurrentHashMap（分段锁，每一个树根节点都有一把锁）
    3. 使用HashTable(全局锁，高并发性能不好)
